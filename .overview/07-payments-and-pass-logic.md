# Payments And Pass Logic

## Payment Rules

- only `STUDENT` applicants require payment
- `ACADEMIC` and `ADMINISTRATIVE` applicants do not pay
- payment requests can only be created for approved student applications

## Payment Creation Paths

### Main business-flow endpoint

`POST /api/admin/applications/[id]/payment-request`

Checks:

- session must be admin
- application must exist
- application must be approved
- applicant type must be student
- semester must exist
- amount must be greater than zero
- no pending or paid request should already exist

### Secondary admin endpoint

`POST /api/admin/payments/create`

This creates a pending payment request for an approved application, but the application-specific endpoint is more aligned with the current business rules because it includes semester handling and student-only validation.

## User Payment Flow

1. User receives payment request
2. User opens payments page
3. User calls `/api/payments/initiate`
4. System creates or reuses invoice number
5. System calls UddoktaPay checkout API
6. UddoktaPay returns `payment_url` and `invoice_id`
7. System stores these in the `Payment` record

## Payment Verification

### Manual verify endpoint

`POST /api/payments/verify`

Behavior:

- user must own the payment
- system verifies invoice against UddoktaPay
- if gateway says `COMPLETED`, payment becomes `PAID`
- method, sender number, transaction ID, and paid time are updated

### Webhook endpoint

`POST /api/webhooks/uddoktapay`

Behavior:

- receives gateway payment payload
- looks up payment by `invoice_id`
- updates local status to `PAID`, `FAILED`, or `PENDING`

## Why The Payment Design Matters

The system does not trust client redirects after payment. Pass activation depends on verified stored payment state, not on the user simply visiting the success page.

## Computed Pass Logic

Implemented in `lib/pass.ts`.

### Pass ID

Generated by:

```text
STH-<year>-<last_6_chars_of_application_id_uppercase>
```

### Pass Activation

`isPassActive` becomes true when:

- application status is `APPROVED`
- and either:
  - applicant is not a student
  - or the applicant has a paid payment

### Pass Issued Date

The pass issued timestamp is:

- `latestPaidPayment.paidAt` if payment exists
- otherwise `application.updatedAt`

### QR Payload

The QR contains:

- `passId`
- `applicationId`
- `userId`
- `fullName`
- `applicantType`
- `route`
- `pickupPoint`
- `issuedAt`

## Important Clarification

ScholarsPass does not persist a separate pass record. The dashboard pass page computes the pass from existing transport and payment data every time it is needed.
